## 一、概述

### 1.1 什么是 Context Engineering

**Context Engineering（上下文工程）** 是结构化设计和管理 AI 模型上下文的技术实践，目的是确保 AI 在多次交互中保持一致性和准确性，避免"Context Rot"（上下文腐烂）。

```
Context Engineering 定义：

┌─────────────────────────────────────────────────────────────┐
│                    Context Engineering                        │
│                                                             │
│  输入：结构化的上下文 Artifact                               │
│  处理：LLM 模型基于完整上下文生成响应                        │
│  输出：跨会话一致、准确的结果                                │
│                                                             │
│  核心目标：                                                  │
│  ├── 让 AI 获得完整项目信息                                 │
│  ├── 让 AI 理解历史决策                                     │
│  ├── 让 AI 保持编码风格一致                                 │
│  ├── 让 AI 避免重复错误                                     │
│  ├── 让 AI 记住用户偏好                                     │
│  └─────────────────────────────────────────────────────────┘
│                                                             │
│  vs Prompt Engineering：                                     │
│  Prompt Engineering = 单次交互优化                           │
│  Context Engineering = 多次交互一致性                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 三大 Engineering 对比

| 工程类型 | 关注点 | 时间范围 | 输入 | 输出 |
|----------|--------|----------|------|------|
| **Prompt Engineering** | 单次提示优化 | 单次交互 | 一个提示 | 一个理想输出 |
| **Context Engineering** | 上下文结构化 | 多次交互 | 结构化 artifact | 跨会话一致输出 |
| **Harness Engineering** | 系统可靠性 | 系统级别 | 流程+验证+安全 | 端到端可靠执行 |

### 2.3 Fresh Context 原理

```
Fresh Context 设计：

┌─────────────────────────────────────────────────────────────┐
│              FRESH CONTEXT PER AGENT/ITERATION              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  传统方式（Context Rot）：                                   │
│                                                             │
│  Session 1 ──────────────────────────────────────► Session N │
│     │                                                    │   │
│     ├─ 历史对话累积                                      │   │
│     ├─ 垃圾信息累积                                      │   │

│     ├─ 质量逐渐下降                                      │   │
│     └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Fresh Context 方式（GSD/Ralph）：                           │
│                                                             │
│  Agent 1 (Fresh) ──► Agent 2 (Fresh) ──► Agent N (Fresh)    │
│     │                │                │                     │
│     ├─ 干净 200K     ├─ 干净 200K     ├─ 干净 200K          │
│     ├─ 无历史对话    ├─ 无历史对话    ├─ 无历史对话          │
│     ├─ 结构化上下文  ├─ 结构化上下文  ├─ 结构化上下文        │
│     ├─ 质量一致      ├─ 质量一致      ├─ 质量一致            │

│     └────────────────┴────────────────┴─────────────────────┘
│                          │                                  │
│                          ↓                                  │
│                    记忆持久化                                │
│                    ├─ Git history                           │
│                    ├─ progress.txt                           │
│                    ├─ prd.json                               │
│                    ├─ AGENTS.md                              │
│                                                             │
│  Fresh Context 核心：                                        │
│  ├── 每次代理：新鲜实例、干净上下文                          │
│  ├── 记忆持久化：文件系统、不依赖对话历史                     │
│  ├── 上下文来源：结构化 artifact、不是对话                   │
│  ├── 代理不疲劳：永远精力充沛                                │
│  └─────────────────────────────────────────────────────────┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
Context Engineering 核心原则：

┌─────────────────────────────────────────────────────────────┐
│                  Context Engineering 原则                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 结构化优于对话                                           │
│     ├── 结构化文档占用更少 token                             │
│     ├── 结构化信息更持久                                     │
│     ├── 对话历史容易腐烂                                     │
│     └─────────────────────────────────────────────────────┘
│                                                             │
│  2. 持久化优于临时                                           │
│     ├── 重要信息写入文件                                     │
│     ├── 文件可跨会话使用                                     │
│     ├── 临时信息保持在对话                                   │
│     └─────────────────────────────────────────────────────┘
│                                                             │
│  3. Fresh Context 优于累积                                   │
│     ├── 每次代理干净上下文                                   │
│     ├── 记忆通过文件持久化                                   │
│     ├── 避免 Context Rot                                    │
│     └─────────────────────────────────────────────────────┘
│                                                             │
│  4. 分层优于混杂                                             │
│     ├── 项目级 → Memory Bank                                │
│     ├── 用户级 → Claude Memory                              │
│     ├── 学习级 → progress.txt                                │
│     ├── 临时级 → 对话                                       │
│     └─────────────────────────────────────────────────────┘
│                                                             │
│  5. 沉淀优于重复                                             │
│     ├── 关键学习写入 AGENTS.md                               │
│     ├── 通用模式写入 Codebase Patterns                       │
│     ├── 避免每次重新解释                                     │
│     └─────────────────────────────────────────────────────┘
│                                                             │
│  6. 按重要性分配                                             │
│     ├── 高价值 → 大 token 空间                               │
│     ├── 低价值 → 小 token 空间                               │
│     ├── 无价值 → 不包含                                     │
│     └─────────────────────────────────────────────────────┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
